Security Platform

Repositorio central de seguridad. Proporciona workflows reutilizables, reglas Semgrep, politicas de gobernanza y templates listos para aplicar en cualquier repositorio de la organizacion.

Que incluye

Workflows reutilizables

  • Semgrep SAST con security gate
  • Dependabot check y alertas
  • SLSA Level 3 provenance

Un cambio, todos los repos

  • Los consumers llaman a este repo
  • Actualizacion automatica en cada run
  • Sin PRs en repos consumer

Incorporacion sin scripts

  • Template repo para repos nuevos
  • 1 archivo para repos existentes
  • Deteccion automatica de nuevos repos

Integraciones

  • GitHub Advanced Security (GHAS)
  • Semgrep Cloud Platform
  • Registro central de excepciones

Arquitectura del sistema

Las reglas y la logica de seguridad viven aqui. Los repos consumer solo tienen un archivo de 15 lineas que llama a este repo. Cuando se actualiza algo aqui, todos los repos lo reciben automaticamente en su siguiente run. 

jgutierrezdtt/
├── security-platform/           ← Reglas, workflows, tutoriales (este repo)
│   └── .github/workflows/reusable/
│       ├── semgrep-scan.yml     ← logica de Semgrep
│       ├── dependabot-check.yml ← logica de Dependabot
│       └── slsa-build.yml       ← firma de artefactos
│
├── security-consumer-template/  ← GitHub Template — para repos nuevos
│   └── .github/workflows/
│       └── security.yml         ← llama a security-platform@main
│
├── security-example-app/        ← Tutorial vivo — ejemplo funcional
│
├── security-exceptions/         ← Registro de excepciones (read-only)
│
└── tu-repo/                     ← Cualquier repo consumer
    └── .github/workflows/
        └── security.yml  ───────► security-platform (reglas aqui)

Por que este modelo resuelve el problema de escala

Si tienes 200 repos y decides cambiar el umbral de severidad de HIGH a CRITICAL, solo cambias un valor en semgrep-scan.yml de este repo. En el siguiente PR de cada repo, el nuevo umbral ya esta activo. No hay PRs que abrir, no hay archivos que sincronizar.

Repos nuevos — un clic, sin configuracion

security-consumer-template es un GitHub Template Repository. Crear un repo desde el activa los checks de seguridad desde el primer commit, sin copiar archivos ni ejecutar comandos.

  1. Ir al template y pulsar “Use this template”

    github.com/jgutierrezdtt/security-consumer-template/generate

  2. Dar nombre al repo y crear

    GitHub copia todos los archivos del template al nuevo repo.

  3. Anadir los secrets (una sola vez)

    gh secret set EXCEPTIONS_READER_TOKEN --repo jgutierrezdtt/tu-nuevo-repo
gh secret set DEPENDABOT_CHECK_TOKEN  --repo jgutierrezdtt/tu-nuevo-repo

  4. Abrir el primer PR — los checks se ejecutan automaticamente

    No hay ningun paso mas. Los cambios futuros en las reglas de seguridad llegan solos.

Repos existentes — anadir un archivo

Para repos que ya existen, el unico paso es crear .github/workflows/security.yml con este contenido: 

name: Security

on:
  pull_request:
  push:
    branches: [main]

permissions:
  contents: read
  security-events: write
  pull-requests: write

jobs:
  semgrep:
    uses: jgutierrezdtt/security-platform/.github/workflows/reusable/semgrep-scan.yml@main
    secrets:
      EXCEPTIONS_READER_TOKEN: ${{ secrets.EXCEPTIONS_READER_TOKEN }}
      SEMGREP_APP_TOKEN: ${{ secrets.SEMGREP_APP_TOKEN }}

  dependabot:
    uses: jgutierrezdtt/security-platform/.github/workflows/reusable/dependabot-check.yml@main
    secrets:
      DEPENDABOT_CHECK_TOKEN: ${{ secrets.DEPENDABOT_CHECK_TOKEN }}

Ese archivo no vuelve a tocarse. Cualquier actualizacion de seguridad se hace en este repo (security-platform) y se propaga automaticamente.

Deteccion de repos sin configurar

El workflow detect-unconfigured-repos.yml se ejecuta cada lunes y crea un GitHub Issue con todos los repos que aun no tienen security.yml. El security team recibe la notificacion sin necesidad de ejecutar ningun comando.

Workflow: semgrep-scan.yml

Escaneo SAST con Semgrep. Descarga excepciones aprobadas del registro central, filtra falsos positivos, genera tabla visual en el PR y aplica security gate configurable por severidad.

InputTipoDefaultDescripcion
scan-scopestringdiffEscanear solo el diff o el repo completo
fail-on-severitystringhighSeveridad minima para fallar el check
upload-sarifbooleantrueSubir resultados al Security tab de GitHub
create-issuesbooleanfalseCrear GitHub Issues por hallazgo
report-onlybooleanfalseGenerar reporte sin bloquear el PR
exceptions-repostringjgutierrezdtt/security-exceptionsRepositorio de excepciones
SecretRequeridoDescripcion
SEMGREP_APP_TOKENOpcionalToken de Semgrep Cloud Platform
EXCEPTIONS_READER_TOKENRequeridoFine-grained PAT read-only al repo de excepciones

Workflow: dependabot-check.yml

Valida el estado de las alertas de Dependabot y bloquea el pipeline si hay vulnerabilidades criticas no resueltas. Genera tabla visual con conteo por severidad y pega un comentario sticky en el PR.

InputDefaultDescripcion
fail-on-criticaltrueFallar si hay alertas CRITICAL abiertas
fail-on-hightrueFallar si hay alertas HIGH abiertas
max-critical0Numero maximo de alertas CRITICAL permitidas
report-onlyfalseSolo reportar sin bloquear

Workflow: slsa-build.yml

Genera provenance SLSA Level 3 para artefactos de release usando el generador oficial de la SLSA Framework con Sigstore/Cosign para firma sin clave privada. Solo se ejecuta en eventos de release o tag.

InputRequeridoDescripcion
artifact-nameSiNombre del artefacto a firmar
artifact-pathSiRuta al artefacto en el runner
build-commandSiComando de build a ejecutar
go-versionOpcionalVersion de Go si el proyecto usa Go
node-versionOpcionalVersion de Node.js si aplica
python-versionOpcionalVersion de Python si aplica
